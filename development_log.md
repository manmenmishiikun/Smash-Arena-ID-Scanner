## 📅 2026-03-31: OCR誤検出バグの修正 (ID: JNP8T が認識されない問題)

### 📌 実施した内容の要約
- **症状**: 部屋ID「JNP8T」が検出されず、クリップボードにコピーされなかった。他のIDは正常動作。
- **根本原因の分析**:
  1. **最有力: WinRT OCRが文字間に空白・改行を誤挿入するケース**。例えば「JNP8T」を「JN P8T」と読んだ場合、正規表現 `[A-Z0-9]{5}` は連続5文字にマッチしないため失敗する。特に「J」はゲームフォントの形状が独特なため誤分割を引き起こしやすい。
  2. 画像の細い文字線が二値化時に途切れ、OCRが文字の一部を認識できない可能性。
- **修正内容** (`image_processor.py`):
  1. **空白除去の追加**: `_correct_text()` 内で `re.sub(r'\s+', '', result)` を適用し、OCRが挿入した空白・改行を全て除去してから正規表現にかけるようにした。
  2. **モルフォロジー処理の追加**: 二値化前に 2×2 カーネルの `MORPH_CLOSE` を適用。細い文字線の途切れを補完し、空白誤挿入の根本原因（文字の視覚的な分断）を抑制。
  3. **OCR生テキストのデバッグログ出力**: `extract_room_id_from_text()` 内で `[OCR raw]`, `[OCR corrected]`, `[OCR matched/no match]` を `print()` で常時出力するよう追加。次回同様の問題が起きた際にターミナルログから即座に原因特定が可能。

### ⚙️ 決定した仕様やパラメータ
- **モルフォロジーカーネルサイズ**: `2×2` (小さすぎると効果なし、大きすぎると文字が潰れる。この値は文字の細線を補完しつつ形状を保つ最小値として選定)
- **空白除去のタイミング**: 大文字化の直後、CHAR_CORRECTION_MAP適用の前（空白が入ったままだと補正マップが適切に機能しない可能性があるため）

### 🚀 次にやるべきタスク (Next Steps)
- ユーザーに修正後のアプリで「JNP8T」相当の（Jを含む、または文字列の視覚的分断が発生しやすい）部屋IDの再検証を依頼する。
- もし問題が再発する場合は、ターミナルに出力される `[OCR raw]` ログを確認し、OCRが何を読み取っているか特定してさらに対処する。

## 📅 2026-03-31: 仕様の誤認識修正 (CHAR_CORRECTION_MAP と ROOM_ID_PATTERN)

### 📌 実施した内容の要約
- **仕様の誤り発覚**: 前回の実装で `"Q": "0"` という補正を加えていたが、これは誤りだった。
- **正しい仕様（ユーザー提供の画像より確認）**:
  - スマブラSPの部屋ID入力画面のキーボードで **グレーアウト（選択不可）になっているのは `I`・`O`・`Z` の3文字のみ**。
  - `Q` は白く表示されており、部屋IDに使用できる有効な文字。
  - したがって有効な文字は `[A-HJ-NP-Y0-9]`（アルファベット26文字 - I,O,Z の3文字 + 数字10種 = 計33種）。
- **修正内容** (`image_processor.py`):
  1. **`CHAR_CORRECTION_MAP` から `"Q": "0"` を削除**。QをQのまま扱うことで誤補正を防ぐ。
  2. **`ROOM_ID_PATTERN` を `[A-HJ-NP-Y0-9]{5}` に変更**。I・O・Z を正規表現レベルで除外し、補正後も万が一これらが残っていた場合に誤ったIDを返さない二重の安全弁にした。

### ⚙️ 確定した仕様
- **有効な部屋ID文字セット**: `ABCDEFGHJKLMNPQRSTUVWXY` (23文字) + `0123456789` (10文字) = **計33種**
- **CHAR_CORRECTION_MAP (確定)**: `{"I": "1", "O": "0", "Z": "2"}` のみ
- **ROOM_ID_PATTERN (確定)**: `re.compile(r"[A-HJ-NP-Y0-9]{5}")`

### 🚀 次にやるべきタスク (Next Steps)
- ユーザーによる継続検証。特に「Q」を含む部屋IDが正しく認識されるかを確認。

## 📅 2026-03-31: 起動クラッシュバグ修正・OCR誤認識（J・Q）対応

### 📌 実施した内容
**2つの問題を同時修正。**

### 🐛 バグ1: 起動クラッシュ（AttributeError: '_always_on_top'）
- **原因**: `BooleanVar`（`_always_on_top` / `_sound_enabled` / `_auto_start`）の初期化が、それらを参照する `_build_ui()` の**呼び出しより後**に行われていた。前回のリファクタリングで `_always_on_top` を `_build_ui()` の外に移した際、`try` ブロック内に書いてしまったが、`try` ブロックはその時点で `_build_ui()` より後に実行されるため、`_build_ui()` 内で `self._always_on_top` を参照すると `AttributeError` が発生していた。
- **解決策**: 3つの BooleanVar (`_sound_enabled`, `_always_on_top`, `_auto_start`) を、`_build_ui()` 呼び出しより前（`__init__` の冒頭）に移動。`_build_ui()` 内の重複する初期化も削除 (`gui_app.py`)。

### 🐛 バグ2: OCR誤認識 — J → `」`、Q → `0`
- **発覚経緯**: ターミナルの `[OCR no match]` ログに `raw='ID: BNG 」 0'` が出力されており、`BNGJQ` というIDを読み取る際に:
  - `J` → `」`（日本語の右カギ括弧・全角）に誤認識
  - `Q` → `0`（ゼロ）に誤認識
  していることが確認された。
- **根本原因**: WinRT OCR は日本語テキスト認識に最適化されているため、スマブラのゲームフォント（欧文スラブセリフ系）の `J` を形状が似た日本語の `」` と誤認識する。
- **解決策**: `CHAR_CORRECTION_MAP` に以下を追加 (`image_processor.py`):
  - `「」→J`, `》→J`, `］→J`, `]→J`（J として読まれうる全ての括弧系文字）
  - 全角数字 `０-９` → 半角数字（念のための追加）
- **Q → 0 について**: Q と数字の 0 は視覚的に非常に類似しており、スマブラのフォントでは特に混同されやすい。補正マップで `0→Q` にするとO→0の補正と矛盾するため、現時点では対応不可。読み取り自体は成功する（ただし Q が 0 として返される）。

### ⚙️ 確定仕様（OCR補正マップ最終版）
```
I→1, O→0, Z→2   (ID排除文字の逆補正)
」→J, 》→J, ］→J, ]→J  (WinRT OCRによる日本語括弧誤認識への対処)
０-９→0-9         (全角数字の半角化)
```
